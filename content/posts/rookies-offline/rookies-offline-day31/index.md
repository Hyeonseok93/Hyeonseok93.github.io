## ① 실시간 진행률 UI (Progress) 엔진 전 모듈 확장 이식
진단 스캔 구동 중 대시보드 화면상에 현재 가동 페이즈, 진행률 백분율, 전송 요청 스케일 지표를 실시간으로 폴링 가시화하기 위한 통합 연동 작업을 완료했습니다.
* **백엔드 Progress 제어 레이어:** `backend/app/services/diagnosis_progress.py` 파일이 인메모리(in-memory) 상태 정보를 중앙 관리(`GET /api/diagnosis/progress`)하며, `progress_reporter.py` 헬퍼가 각 모듈의 생명주기에 개입하여 `prepare`, `phase`, `endpoint_progress`, `step_progress`, `zap_phase` 단계를 정밀하게 쪼갭니다.
* **프론트엔드 연동 규칙:** `frontend/src/components/DiagnosisPage.tsx` 컴포넌트가 실행 중 상태를 폴링하여 화면에 표시합니다. `1-5, 3-2, 3-4, 3-5, 3-6, 4-1, 4-2` 모듈은 공통 헬퍼 클래스를 이식해 표준 진행률을 보장하며, 대량의 본문 연산 및 퍼징 사격이 수반되는 `5-2, 6-1` 모듈은 성능 최적화를 위해 내부 페이즈(`httpx_pii` / `httpx_fuzz`) 단에서 서비스를 직접 제어하도록 예외 설계했습니다.

## ② 로컬 vs Docker 환경 Probe URL 라우팅 규칙 통일
* **결함 원인:** 백엔드 도커 컨테이너 내부망 환경에서 `localhost:8080` 타깃 주소로 통신을 시도할 경우, 외부 호스트 서버가 아닌 컨테이너 자기 자신(Loopback)으로 라우팅되어 스캔 트래픽이 전무 유실되고 body 수집 건수가 0건으로 드롭되는 인프라 오설정을 해결하기 위한 조치입니다.
* **통일 규칙:** 호스트 OS에서 소스코드를 직접 독립 구동할 때는 `localhost` 주소 규칙을 투명하게 유지하되, Docker Compose 컨테이너 구동 시에는 환경 변수 `ARGUS_PROBE_HOST=host.docker.internal`을 감지하여 실제 전송 네트워크 단에서 대역을 동적으로 강제 rewrite 변경하도록 정비했습니다. 관련 연동부는 `zap_util.py` (probe_url 함수), `verify_service.py` 및 각 모듈의 `targets.py` (`probe_base_url` 함수)에 일괄 적용되었습니다. 대시보드 화면 및 결과 리포트 명세 상에는 `localhost` 가시성을 그대로 유지하여 가독성을 보완합니다.

# 2. 입/출력 값 검증 부재 영역 (1-5)

## ── 1-5. 검증되지 않은 리다이렉트와 포워드 모듈
* **보완 조치 및 설계 명세:** 외부 임의의 URL로 열리는 오픈 리다이렉트(Open Redirect), CORS 오설정으로 인한 SOP 무력화 위협 및 `crossdomain.xml` 기밀 노출 리스크를 추적합니다. 아르고스 고속 파이프라인 엔진 내에 `targets.py`와 `redirect_rules.py` 규칙 컴포넌트를 설계하여 판정을 자동화했습니다. `probes.py`가 httpx 실행을 전담하며 `zap_scan.py`가 보조 역할을 수행합니다.
* **동적 검증 및 동작 흐름:** Base URL 및 인벤토리 트리 분석을 수행한 뒤, **Phase A** 트랙에서 오픈 리다이렉트 유발 가능성이 농후한 쿼리 파라미터를 추적하여 사전에 조율된 외부 싱크 목적지 주소(`redirect_sink_base`)로 전송을 변조 감행합니다. 이어서 Origin 헤더를 임의의 해커 사이트로 조작 투사하는 CORS 프로브 및 크로스 도메인 자산 직접 조회를 동시 전개하며, 선택적으로 ZAP 동적 스캔 규칙(40031 / 10028)을 supplemental 연동 트리거하여 차단 여부를 교차 검증합니다. 대시보드 모달 옵션을 통해 수집 범위(`probe_mode`), `check_request_url`, `check_request_body` 등 상세 스케일을 오버라이드할 수 있습니다.

# 3. 취약한 접근 통제 관리 영역 (3-2, 3-4, 3-5, 3-6)

## ── 3-2. 인증 실패 횟수 제한 모듈
* **보완 조치 및 설계 명세:** 무차별 대입 공격(Brute Force) 차단을 위해 연속 패스워드 실패 시 발동되는 임계치 잠금 정책 및 계정 보호(Rate Limit) 가드의 유무를 자동 판정합니다 (응답 메시지의 구조적 동일성만 대조하는 6-2 오류 처리 모듈과는 완전히 분리되어 작동합니다). `lockout_rules.py` 검증기가 핵심 판정을 담당합니다.
* **동적 검증 및 동작 흐름:** `login_discovery_service` 엔진이 추출해 낸 로그인 엔드포인트를 기반으로 가상 테스트 어카운트를 매칭한 뒤, 틀린 패스워드 조합 인자(`wrong_password`)를 기본 12회 연속 강제 전송(`max_attempts`)합니다. 이후 검증기가 HTTP 429 Too Many Requests, 423 Locked 또는 403 Forbidden 상태 코드 전환 여부를 체크하고 `Retry-After` 헤더 배출 및 오류 텍스트 바디 컨텍스트의 실시간 변화 추이를 연산하여 잠금 정책의 적합성을 판정합니다. 프론트엔드 옵션 패널에서 시도 임계치를 3~25회 사이로 오버라이드 조율할 수 있습니다.

## ── 3-4. 관리자 페이지 분리 여부 모듈
* **보완 조치 및 설계 명세:** 최고 관리자 및 백오피스 거버넌스 채널이 일반 사용자 대외 서비스 대역과 호스트, 서브도메인, 로그인 URL, 추측 가능 경로 수준에서 안전하게 격리되어 보호받고 있는지 구조적 분석을 집행합니다. `targets.py`가 인벤토리 및 관리자 호스트 목록을 로드합니다.
* **동적 검증 및 동작 흐름:** 인프라 네트워크 트래픽 부담을 덜기 위해 **실제 HTTP 네트워크 패킷을 단 1건도 전송하지 않는 무증적 정적 분석 파이프라인**을 개척했습니다. `api-tree` 인벤토리 데이터 구조, 배포 프로필별 도메인 맵 변수 규칙, 그리고 `verify-report.json` 로그 명세와 대시보드 Base URLs 상태를 상호 결합 로드하여 `separation_rules.py` 규칙 엔진이 분석을 대행합니다. 사용자 도메인과 어드민 도메인이 동일한 호스트 세션 주소 체계를 공유하거나 예측하기 쉬운 레거시 주소 구문(`/admin` 등)이 인덱싱되어 발견되면 즉각 경고 및 정보 유출 위험 지표를 발출합니다. 프론트엔드에서는 별도 옵션창 없이 원클릭으로 구동되며 finding은 `inventory` / `info` 섹션에 기본 펼침 제공됩니다.

## ── 3-5. 검색엔진 정보 노출 가능성 모듈
* **보완 조치 및 설계 명세:** Google Dorking 및 외부 웹 크롤러 로봇에 의해 비공개 포탈 주소가 무단 색인 누출되는 현상을 제어하는 검색엔진 차단 거버넌스 인벤토리를 수집합니다. `targets.py`가 프론트엔드 및 API 베이스 주소의 중복을 제거하고 처리합니다.
* **동적 검증 및 동작 흐름:** 권한이 없는 익명 패스(`anonymous pass`) 상태에서 프론트엔드 기본 자산 도메인 뒤에 `/robots.txt` 경로를 강제 호출하여 `Allow/Disallow` 구문 명세를 파싱합니다. 이어서 테스트 가상 계정 세션별 토큰을 의무 인젝션 바인딩한 인증 패스(`authenticated pass`) 상태로 동일 페이지의 자바스크립트 렌더링 HTML 소스코드를 추적 리드하여 head 대역 내에 `<meta name="robots" content="noindex, nofollow">` 메타 수집 거부 가드가 유효하게 삽입 작동하고 있는지 `robots_rules.py`를 통해 동적 추출 및 파싱을 감행합니다.

## ── 3-6. 백업 파일 및 테스트 파일 존재 여부 모듈
* **보완 조치 및 설계 명세:** 형상 관리 및 빌드 배포 과정에서 누락 유실되어 남겨진 은닉 압축 백업본 및 기밀 환경 설정 자산이 대외 유출되는 결함을 제어하기 위해 `file_rules.py` 규칙 엔진을 구성했습니다.
* **동적 검증 및 동작 흐름:** 공통 자산화된 `backup-test-files.txt` 사전에 기재된 확장자 및 파일 명세 코드 배열을 Base URLs 및 `api-tree` 상위 경로 세그먼트와 조합한 뒤, 익명 및 인증 세션별 크로스 대입 스캔을 총괄 집행합니다. 수신 응답을 대상으로 단순히 404/400 차단 텍스트를 리턴한 것인지, 혹은 실제 내부 파일 구조 데이터 가독 환경이 열려 `.bak`, `.old`, `.env`, `web.config` 기밀 정보가 스트리밍 유출되는 상태인지를 본문 바이너리 휴리스틱 필터로 정밀 분류 판정합니다.

# 4. 취약한 인증 및 세션 관리 영역 (4-1, 4-2)

## ── 4-1. 쿠키·웹 스토리지 조작 가능성 모듈
* **보완 조치 및 설계 명세:** 로그인 및 민감 행위 처리 시 배출되는 인증 컨텍스트 속성 위배 상태와 수평적 권한 우회 가능성을 정밀 마스터 체크합니다. `cookie_attr_rules.py` 및 핵심 변조 프로필 모듈이 유기적으로 결합되어 구동됩니다.
* **동적 검증 및 동작 흐름:** 총 3대 샌드박스 파이프라인으로 크로스 체크를 집행합니다.
  * **Phase A (cookie_attr):** 로그인 및 토큰 갱신 API 성공 응답 시 서버가 배출하는 `Set-Cookie` 헤더 속성 내 `HttpOnly`, `Secure`, `SameSite=Lax/Strict` 설정 무결성을 정적 추적 분석합니다.
  * **Phase B (cross_cookie):** 세션 컨텍스트 가로채기 위협 강도를 실증하기 위해, 테스트 계정 A 세션의 유효 쿠키 인자를 가로채 강제로 타인 자산인 테스트 계정 B의 API 엔드포인트 파라미터 영역에 오염 주입 후 전송하여 타인의 정보가 도용 무단 유출(`cross leak`)되는지 확인합니다.
  * **Phase C (tamper):** `Authorization` 베어러 토큰 및 일반 쿠키 문자열 값의 식별 고유 부위를 임의 변조하여 전송 시, 백엔드 필터가 위변조 상태를 명확히 인지하고 거부하는지 변조 무결성을 판정합니다.

## ── 4-2. 인증(세션·토큰) 값 안전성 모듈
* **보완 조치 및 설계 명세:** JWT 및 인메모리 Opaque 토큰 자산의 정적 암호 강도 판정과 실제 구동 애플리케이션 라이프사이클의 실시간 세션 만료 제어 적합성을 검증하기 위해 `token_analyzer.py`와 `lifecycle_probes.py` 컴포넌트를 이식했습니다. 
* ** 중대 내부 크래시 패치 완료:** 실시간 진행률 연산식인 `lifecycle_steps` 코드가 가상 인증 계정을 로드하는 헬퍼인 `pick_probe_account()` 실행 라인보다 먼저 코드 단 위쪽에서 참조되어 참조 오류(`UnboundLocalError`)와 함께 500 내부 에러 크래시를 유발하던 치명적인 소스코드 오염 상태를 발견하여, 금일 자 긴급 수정 조치 트랙을 전개해 계정 컨텍스트 해석 구문을 진행률 팩터 계산식보다 **물리적으로 최전방 상단 레이어로 이동 전치 배치**하여 크래시 버그 현상을 완벽하게 해소했습니다.
* **동적 검증 및 동작 흐름:** 암호 알고리즘 규칙 검증(`JWT alg none/weak check`), 만료 스탬프(`exp`) 바인딩 유무를 연산한 뒤, 동일 세션 재로그인 시 기존 토큰이 즉각 영구 파기되는지, 동일 토큰을 이기종 브라우저 및 크로스 IP 환경에서 다중 중복 동시 접속 시 세션 킬링(`Session Killing`) 가드가 작동하는지, 서버 로그아웃 수신 즉시 Redis 블랙리스트 무효화가 집행되는지 가상 세션 제어 시뮬레이션을 전개합니다.

# 5. 중요 정보 저장/전송 처리 미흡 & 부적절한 오류 처리 영역 (5-2, 6-1)

## ── 5-2. 요청/응답 주요정보 포함 여부 모듈
* **보완 조치 및 설계 명세:** 데이터 직렬화 및 통신 트래픽 패킷 전송 구간 상에서 개인식별정보(PII) 데이터가 마스킹 필터 없이 원문 평문 노출되는 결함을 전수 식별하기 위해 `pii_rules.py` 규칙 엔진을 구성했습니다.
* **한국어 실명(이름) 오탐 방지 가이드라인 규칙 최종 마감:** 차량 매물 정보나 브랜드 카탈로그 상품명(예: '스타리아') 등이 이름 속성으로 오염되어 보안 결함으로 오탐되는 런타임 유해 패턴을 제어하고자 정밀 제약을 완료했습니다. 반드시 `name`, `nickname`, `insuredName` 등 **인간 식별 목적의 성격이 명확한 필드명 및 JSON 명세 경로**여야만 가동하며, `modelName`, `cars[].name` 등의 상품 설명 필드는 필드명 화이트리스트 검사 계층에서 안전하게 스킵 배제합니다. 수신 값이 정확히 한글 3글자(`^[가-힣]{3}$`)를 만족하고 첫 글자가 약 150여 개 자산의 성씨 배열(`_KOREAN_SURNAMES`)에 매칭될 때만 실명 유출로 확정합니다. 이에 따라 브랜드성 가명 계정 및 2글자/4글자 닉네임 등은 무결성을 위해 의도적으로 미탐 격리 처리하도록 설계했습니다.
* **동적 검증 및 동작 흐름:** 대시보드 UI 화면단에 `g52DiagnosisOptions.ts` 명세와 매핑 카운터를 적용하고, 스캔 범위 조율용 프리셋 탭을 완비했습니다. 인벤토리 `api-tree` 내 엔드포인트 중 중복을 제거한 뒤, 익명 접근 및 5개 가상 테스트 계정 권한 조합(`6 auth pass`)을 입혀 트래픽을 송수신하며 반환 JSON 텍스트 트리 구조 전체를 순회 스캔(`JSON walk`)하여 이메일(`email_plain`), 연락처(`phone_plain`), 주민번호(`rrn_plain`), 여권·카드·계좌 및 경로 정보 노출을 추적합니다. 실명이 배제된 기본 `onde-pilot` 실행 시 이메일/연락처 지표만 검출되고 이름 항목은 0건으로 배출되는 것이 정상적인 하드닝 형상입니다.

## ── 6-1. 오류페이지를 통한 정보 노출 모듈
* **보완 조치 및 설계 명세:** 입력값 크기 및 자료형 변조 타격을 가해 서버 내부 비즈니스 런타임 크래시를 인위적으로 유도한 뒤, 반사 출력되는 응답 본문 내에서 기밀 자산 지표가 유출되는지 검증하기 위해 `payloads.py`와 `error_rules.py` 판정 엔진을 구성했습니다. `triggers.py`가 엔드포인트별 trigger를 선택 부여합니다.
* **동적 검증 및 동작 흐름:** 프론트엔드 옵션 다이얼로그(`g61`) 및 프리셋 탭 제어 컴포넌트를 이식 완료했습니다. 수집된 인벤토리 샘플 주소를 대상으로 파라미터, 바디 JSON, 경로 변수 영역에 자료형 오염 파라미터(`type confusion`), 대용량 바이너리 더미, 비정형 특수문자 페이로드(`malformed payload`)를 강제 사격 주입합니다. 이후 판정 엔진이 작동하여 응답 본문 내에 Java 스택 트레이스 명세 구문(`java.lang.`), SQL 구문 오류 흔적 및 루트 디렉터리 노출 경로 규칙 단어가 은닉 포착되어 누출되는지 감시하며, ZAP unified fuzz 스캐닝 엔진 정책과 하이브리드 연동 가동합니다.

# 6. 결론 및 Next Milestone: 팀 모듈 통합 및 취약점 동적 증적 자동화 기능 지속 개발

이번 스프린트에서는 SK쉴더스 취약점 진단 가이드라인 표준을 기반으로 아르고스(Argus) 자동 진단 플랫폼의 9대 신규 확장 모듈 프레임워크를 수립하고, 연동 테스트 및 크래시 수정을 완전히 성공시켰습니다. 

이것으로 전사 자동화 탐지 모듈 아키텍처의 1차적인 단독 백엔드 기능 다지기가 일단락되었으며, 우리 팀의 다음 넥스트 스텝 마일스톤은 본격적인 전체 시스템 통합 및 연출력 하드닝 공정으로 신속히 선회합니다.

  1. **전사 도메인 모듈 통합 통합:** 오늘 자로 구현 및 단독 검증 정비가 완료된 본 9대 신규 확장 모듈들과, 팀원들이 각 서브 도메인 영역별로 열심히 만들고 있는 나머지 잔여 모듈 자산 명세들을 **내일 즉각 하나의 코어 파이프라인 브랜치로 일제히 머지(Merge)하여 통합**을 전격 단행할 예정입니다.
