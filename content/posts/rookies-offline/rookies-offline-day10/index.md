# 서론

> **"취약점을 제대로 진단하기 위해서는 먼저 공격이 어떤 방식으로 이루어지는지 명확히 이해해야 합니다. '공격을 알아야 방어가 가능하다'는 본질에 집중하여, 본격적인 진단 페이즈에 앞서 웹 해킹의 구체적인 공격 메커니즘을 파악하고 대표적인 치명적 결함인 SQL Injection의 인증 우회 기법을 선제적으로 학습했습니다."**
>
> 오늘은 인프런 강의를 통해 웹 해킹의 공격 경로와 Burp Suite 프록시 도구 사용법을 익히고, SQL Injection 인증 우회 매커니즘의 기초를 정립했습니다.

# 1. 웹 해킹의 본질과 패러다임의 변화

웹 서비스 상에서 발생하는 보안 허점을 이용하는 **웹 해킹**을 방어하기 위해서는, 공격자가 시스템에 어떻게 침투하는지 그 방식부터 명확히 파악해야 합니다. 인프런 수업을 통해 웹 해킹의 이론적 기초와 위협 경로를 학습했습니다.

* **웹 서비스가 주요 공격 경로인 이유:** 네트워크 방화벽은 대외 서비스를 위해 HTTP(80), HTTPS(443) 포트를 열어둘 수밖에 없습니다. 공격자 입장에서는 이 열려 있는 웹 서비스가 내부망으로 진입할 수 있는 가장 유일한 루트가 됩니다.
* **웹 해킹이 발생하는 주요 원인:**
  1. 입력값 검증 부재 (SQL Injection, XSS 등)
  2. 취약한 인증 및 접근 통제 (권한 상승, 우회)
  3. 중요 정보 및 에러 메시지 과도 노출
  4. 서버 및 프레임워크 자체의 설정 오류 및 설계 결함

# 2. 웹 해킹 필수 도구: 웹 프록시(Web Proxy)와 Burp Suite

>  **"웹 브라우저의 검증을 과신하지 말고, 오직 프록시를 거쳐 나가는 요청과 응답을 검증해야 한다."**

클라이언트와 서버 사이에서 패킷을 중계하는 **웹 프록시**의 매커니즘을 학습했습니다. 브라우저에서 자바스크립트로 아무리 입력값을 제한하더라도, 프록시 툴로 패킷을 가로채면 무력화될 수 있음을 이해하고 실습 도구인 **Burp Suite**의 기본 기능을 익혔습니다.

* **Proxy (Intercept):** 브라우저와 서버 간의 패킷을 중간에 가로채서(Intercept) 사용자가 직접 요청 값을 조작하는 기능
* **Repeater (`Ctrl + R`):** 가로챈 패킷의 입력값을 자유롭게 수정하여 서버의 반응을 반복적으로 테스트하고 재전송하는 기능
* **기타 모듈:** 무작위 대입을 위한 Intruder 및 인코딩/디코딩을 지원하는 Decoder 등

# 3. 기본 개념 이해

강의 가이드에 따라 Apache, PHP, MySQL 기반의 로컬 실습 환경(APM)을 다루며 웹 서버와 데이터베이스 간의 연동 방식을 파악했습니다. 

* **실습 데이터베이스 분석:** 강의에서 제공된 취약점 테스트용 데이터베이스 구조를 살펴보며, 웹 애플리케이션에서 입력값이 검증 없이 쿼리문과 결합할 때 발생하는 위험성을 분석했습니다.
* **Magic Quotes 설정의 이해:** 과거에 싱글 쿼터(`'`) 등을 자동 이스케이프(`\'`) 처리하여 SQL Injection을 방어하려던 `magic_quotes_gpc` 설정의 개념과, 현대 보안 환경에서의 한계점을 학습했습니다.

# 4. SQL Injection 매커니즘 및 인증 우회 원리 분석

SQL Injection은 사용자 입력값이 적절한 필터링 없이 개발자가 작성한 미완성 SQL 구문과 문자열로 결합(Concatenation)되면서, 데이터베이스 엔진이 이를 명령어로 오인하여 실행할 때 발생합니다.

## ① 로그인 페이지 인증 우회 (Authentication Bypass) 원리

* **타깃 계정의 아이디를 알고 있는 경우 (주석 활용):**
  * 아이디 입력창에 `admin'#` 또는 `admin' -- `와 같은 구문을 주입하면, 서버 내부 쿼리에서 패스워드를 검증하는 뒷부분(`AND pw='...'`)이 통째로 주석 처리되어 비밀번호를 몰라도 로그인이 성공하는 원리입니다.
* **아이디를 모르는 경우 (항상 참인 조건 주입):**
  * 입력창에 `' or 1=1#`을 주입하면, `WHERE id='' OR 1=1` 구조가 되어 전체 조건식이 무조건 참(True)이 됩니다. 이 경우 데이터베이스 결과셋의 가장 최상위 레코드(보통 관리자 계정)로 인증이 우회되는 메커니즘을 배웠습니다.

## ② 특정 인증 페이지 및 비밀글 우회 원리 (In-line & Terminating)
게시글 조회나 비밀번호 확인 창에서 식별자(`idx`)가 지정되어 있더라도, 패스워드 입력란에 `' or '1'='1` (인라인 방식) 또는 `' or 1=1#` (터미네이팅 방식)을 주입하면 연산자 우선순위(AND가 OR보다 먼저 연산됨)에 의해 전체 쿼리가 참으로 왜곡되어 인증을 우회할 수 있음을 파악했습니다.

# 5. 오늘의 주요 학습 성과

1. **공격자 관점의 선행 학습:** 무작정 진단을 수행하기 전, 취약점이 발생하는 근본적인 공격 원리와 패킷 조작 방식을 명확히 이해함.
2. **프록시 도구 숙달:** Burp Suite의 Intercept 및 Repeater 기능을 활용한 패킷 제어 흐름 체화.
3. **SQL Injection 기초 수립:** 인라인 및 주석 종료 방식을 이용한 논리 구조 변조 및 인증 우회 매커니즘 파악.

# 6. Next Step: SQLi 심화 및 차세대 웹 취약점 학습 로드맵

인프런 강의 진도에 맞춰, 향후 진단 및 방어 역량 내재화를 위해 아래 항목들을 순차적으로 학습할 예정입니다.

1. **SQL Injection 심화:** 데이터 병합을 이용한 `UNION-Based` 공격, 참/거짓 반응 기반의 `BLIND-Based` 데이터 탈취 및 Prepared Statement 기반 시큐어 코딩 실습
2. **OS Command Injection:** 웹 서버 운영체제 명령어를 주입하는 원리 분석 및 Reverse-Shell 공격과 방어 적용
3. **XXE Injection:** XML 외부 엔티티 참조 취약점 원리 파악 및 취약 환경 시큐어 코딩
4. **XSS (Cross Site Scripting):** DOM/Reflected/Stored XSS 공격 유형 분석 및 세션 하이재킹 대응 실습
5. **CSRF (Cross Site Request Forgery):** 사이트 간 요청 위조 원리 분석 및 CSRF Token 동작 원리 상세 실습
