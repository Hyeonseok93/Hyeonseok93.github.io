# 서론

> **"이전 스키마·연동 작업을 이어 온데(onde)를 진단 타깃으로 연결한 뒤, 온데를 스캔할 '웹 취약점 자동 진단 플랫폼'을 기획했습니다. 오픈소스(ZAP·Semgrep)를 분석한 뒤 AI 우선순위 분류와 Selenium 자동 증적 캡처를 포함한 3개 핵심 MVP 기능을 정했습니다."**
>
> 온데 타깃 연동을 정리한 뒤, 이를 스캔할 웹 취약점 자동 진단 플랫폼 기획으로 넘어가 ZAP·Semgrep 기반 3대 MVP 기능을 확정했습니다.

# 1. 온데(onde) 타깃 연결과 자동 진단 기획으로의 전환

이전 구간의 스키마·연동 작업을 이어, 오늘부터는 온데를 취약점 진단 대상(Target)으로 연결하고 **웹 취약점 자동 진단 플랫폼** 기획으로 넘어갔습니다. (스키마 V1/V5·Flyway·LBS 등 상세는 Day 12 기록을 참고합니다.)

# 2. 웹 취약점 자동 진단 플랫폼 기획 방향과 성격 정리

우리 팀 **'너의 취약점은'**은 웹 서비스의 보안 결함을 자동으로 찾아 추적하는 플랫폼을 만들기 위해 **보안 진단 플랫폼의 방향과 진단 대상을 분리해 정리**했습니다.

- **진단 대상과 진단 플랫폼의 명확한 이원화 구도 정립:**
  - 우리가 개발한 개선된 여행 플랫폼 **'온데(onde)'는 취약점 진단의 대상(Target)**이 됩니다.
  - 이와 동시에 온데의 취약점을 자동으로 스캔하고 관리자에게 시큐어 코딩 보완 내용을 보여 주는 **'웹 취약점 자동 진단 플랫폼'의 핵심 구조를 연결해 함께 만듭니다.**
- **범용 웹 진단 플랫폼으로 확장:** 특정 도메인 전용 도구가 아니라 여러 웹 서비스 환경을 스캔할 수 있는 **'범용 진단 플랫폼'**으로 방향을 정했습니다.
- **기획 구성 비율(50:50) 정의:** 기술 문제를 다루는 **'기술 논문 형식' 50%**와, 관리자가 보기 편한 **'서비스 형식' 50%** 비율을 정했습니다.
- **주제 개요와 필요성 정리:** 이 플랫폼이 필요한 이유를 정리하고, 시나리오 기반 취약점 탐색 과정을 설계했습니다.

# 3. 차별점 확보를 위한 오픈소스 도구 분석 및 3대 핵심 MVP 기능 도출

기존 단일 스캐너와 다른 기능을 만들기 위해 현업에서 쓰는 오픈소스 진단 도구를 조사하고, 이를 함께 제어하는 파이프라인과 **3개 핵심 기능**을 골라 MVP 사양을 설계했습니다.

## ① 오픈소스 진단 툴 리서치 및 라인업 확정

- **OWASP ZAP:** 동적 애플리케이션 보안 테스트(DAST) 엔진으로 채택하여 구동 중인 웹 서비스를 대상으로 실시간 패킷 퍼징 및 주입 공격 취약점 자동 스캔.
- **Semgrep:** 정적 애플리케이션 보안 테스트(SAST) 소스코드 분석 도구로 채택하여 개발 단계의 소스코드 내 시큐어 코딩 규칙 위반 여부 스캔.
- **SSL Labs:** 외부 통신망의 안전성을 보장하기 위해 웹 서버의 SSL/TLS 암호화 프로토콜 설정 및 인증서 취약점 평가.

## ② 플랫폼 3개 메인 핵심 기능 및 MVP 스펙

1. **AI 기반 조치 우선순위 분류 알고리즘:** 대량의 로우 데이터(Raw Data) 취약점이 발견되었을 때, 인공지능 API와 프롬프트 설계를 통해 어떤 결함을 먼저 보완해야 하는지 위험도 및 시급성별로 취약점을 자동 분류합니다.
2. **Selenium 기반 스크린 자동 캡처:** 인프라 트랙의 윈도우 서버(Windows Server) 환경에서 헤드리스(Headless) 웹 드라이버 실행 스크립트를 구현해, 자동 진단 과정에서 취약점이 매핑되는 순간의 증적 화면을 자동 캡처해 저장합니다.
3. **커스텀 한글 보안 진단 리포트 엔진:** 스캔된 통계 지표와 AI가 바인딩한 시큐어 코딩 대응 가이드를 결합하여, 가독성 높은 고품질의 표준 한글 보고서(PDF)를 원클릭으로 자동 생성 및 출력합니다.

# 4. 저장소 문서화 최신화 및 가상화 서버 스크립트 빌드 테스트 현황

실전 개발 및 플랫폼 기획 구체화와 동시에, 팀의 협업 표준화 유지를 위한 레포지토리 관리 및 인프라 사전 검증 작업을 수행했습니다.

- **백엔드(Backend) README.md 수정 보완:** 13일차에 완료된 메인 통합 소스코드의 엔드포인트 구조와 최신 시스템 컴포넌트 사양을 백엔드 공식 기술 명세서에 반영했습니다.
- **인프라(Infra) 사양 가이드라인 수정 반영:** 클라우드 구성 토폴로지와 컨테이너 가상화 배포 사양을 인프라 공식 README 문서에 녹여내는 가이드라인 수정 작업을 활발히 진행 중입니다.
- **Selenium 캡처 환경 스크립트 테스트 (인프라):** 가상화 서버에서 크롬 드라이버와 맞춰 헤드리스로 페이지를 열고 스크린샷을 파일로 남길 수 있는지 사전 스크립트·구동 테스트를 진행 중입니다.
- **보안 진단 학습:** 인프런(Inflearn) 웹 보안·시큐어 코딩 과정을 끝까지 듣고, 이후 플랫폼 결함을 볼 때 쓸 이론 정리를 마쳤습니다.

# 5. 다음 단계: 진단 파이프라인 코딩 및 크롤러 스크립트 연결

이기종 보안 엔진을 제어할 시스템 아키텍처와 AI 매핑 규격이 수립되었으므로, 다음 마일스톤부터는 설계를 실제 소스코드로 구현하는 본격적인 빌드 단계로 들어갑니다.

- **오픈소스 제어 파이프라인 구현:** 백엔드 서버 내에 ZAP API 호출 클라이언트 모듈 및 Semgrep CLI 결과 파일 파서 클래스 코드 작성 완료.
- **Selenium 크롤러 실전 결합:** 윈도우 서버 인프라 환경에서 검증 중인 Selenium 헤드리스 자동 캡처 스크립트를 플랫폼 백엔드 이벤트 핸들러와 물리적으로 연동.
- **AI 응답 데이터 파싱 모듈 테스트:** 프롬프트 매핑 템플릿을 통해 인입되는 LLM API의 JSON 응답 데이터를 자바 엔티티 객체로 변환하여 데이터베이스 무결성을 검증하는 통신 테스트 진행.
