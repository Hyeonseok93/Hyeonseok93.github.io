# 2. 2차 주간 멘토링 보고: 온데(Onde) 플랫폼 점검 및 아르고스 3단계 사양 확정

장대현 멘토님께 클라우드에 배포 완료된 온데(Onde) 여행 플랫폼 화면을 직접 시연해 드리고 시스템 구조도(`Onde_Architecture.pdf`)를 바탕으로 심층 점검을 받았습니다. 아울러, 취약점 진단 플랫폼 아르고스의 계획을 공유하고 3단계 코어 파이프라인 스펙을 확정했습니다.

### ① 진단 대상 '온데(Onde)' 플랫폼 기능 고도화 및 취약점 격리 주입

알파 테스트 결과 및 인프라 구성도를 보완하기 위해 각 도메인별 기능 확장과 취약점 테스트베드 주입 피드백을 수령했습니다.

- **사용자(User) 도메인 확장:** 여행기 리뷰 내에 '비밀 댓글 기능'과 '여행기 수정 기능'을 신설하고, 개인정보(전화번호 수정, 비밀번호 변경 등) 마이페이지 프로필 관리 기능을 추가 반영했습니다.
- **판매자(Seller) 권한 및 대시보드 정비:** 판매자 기능은 `Seller Admin`에서 일괄 수행할 수 있도록 권한 부여 체계를 확립합니다. 정산 승인 관리 화면에서 총 매출액뿐만 아니라 상세 내역을 파악할 수 있는 '판매자 대시보드'와 판매자가 상품 값을 임의 조작하지 못하도록 고정 단가 대비 할인율만 조정할 수 있는 비즈니스 로직을 구축합니다. 판매자가 관리자에게 파트너 신청 후 최종 승인이 떨어지기 전 파라미터를 조작하여 임의 승인 상태로 변조시키는 취약점과 국세청 사업자 등록 번호 진위 확인 버튼 우회 결함도 검증용으로 유지합니다.
- **관리자(Admin) 거버넌스 및 도메인 분리:** 보안성 확보를 위해 사용자 로그인 창과 관리자 로그인 창이 결합된 구조를 폐기하고 **로그인 페이지를 완전히 분리**하며 IP 접근 제어 가드 및 2차 인증(2FA)을 기획에 반영했습니다. 관리자 패널에서 가입 사용자의 상세 정보(이름 등)가 가시성 있게 모니터링되도록 보완하고, 서버가 분리된 만큼 사용자 도메인과 관리자 도메인 2개로 이원화하는 방안을 확정했습니다.
- **온데 취약점 정밀 맵 설계:** 고정 가격 상태에서 투숙객 수와 객실 수를 조작하여 최종 결제 단가를 위조하는 **금액 및 파라미터 변조 취약점**, **마일리지 무단 조작 결함**, 타 사용자의 식별자(ID) 조작을 통해 마이페이지 정보에 무단 접근할 수 있는 **IDOR 취약점**을 배치합니다. PDF 출력 기능 등에서 발생하는 단순 경로 조작 행위는 제외하되, 실제 패킷 덤프 기반으로 추적 가능한 **SSRF, 파일 인클루전(File Inclusion), 중요 정보 파일 다운로드 취약점**을 명확히 심어둡니다. 또한 메인 검색창 등의 입력 데이터 처리부에는 **Reflected XSS** 취약점이 성립되도록 열어두고 데이터 마스킹 필터를 연계 검증하기로 했습니다.

### ② 아르고스(Argus) 플랫폼 계획 및 3단계 상세 명세

- **STEP 01. 탐지 및 AI 우선순위 산정:** 전방위적 보안 취약점을 스캔함과 동시에 AI가 비즈니스 맥락을 분석해 우선순위를 판별합니다. OWASP ZAP(동적), Semgrep(정적), SSL Labs(인프라)를 이용해 결함 로그를 일괄 수집합니다.
- **STEP 02. Selenium 증거 캡처:** 1단계에서 우선순위가 정해진 주요 취약점들을 시각적 증거로 캡처하여 개발자에게 직관적으로 제공합니다. 취약점이 실제 위협으로 이어지는지 확인하기 위해 **Selenium Headless 브라우저가 모의 공격 시나리오를 자동으로 실행(Attack Replay)**하며, XSS 공격으로 경고창이 팝업되거나 비정상 페이지로 리다이렉트되는 순간 등의 브라우저 렌더링 화면을 캡처하여 시각 자료를 확보합니다.
- **STEP 03. 조치 가이드 및 리포팅:** 획득된 취약점 데이터와 스크린샷 증적을 종합하여 **SK쉴더스 Web/API 취약점 진단 가이드 항목(8개 그룹 28개 항목)**과 탐지된 취약점을 1:1로 매핑하여 체계적으로 분류합니다. Java, Python, PHP 등 언어 및 프레임워크별 구체적인 코드 수정 예시를 제공하며, 기술팀용 상세 증적 자료와 경영진용 종합 요약이 모두 포함된 SK쉴더스 대응 한글 PDF 보고서(사용자/판매자 보고서, 관리자 보고서 총 2종)를 원클릭으로 최종 발행합니다.


# 3. [취약점 진단 특강 강의] 실전 모의해킹 방법론 핵심 요약

대면 멘토링 세션 중 장대현 멘토님께서 진행해 주신 실전 보안 진단 필드의 방법론, 증적 확보 기준, 컴플라이언스 체계에 대한 집중 강의 내용을 요약 정돈했습니다.

- **동적 진단 중심 및 정적 키워드 추적:** 실제 웹/API 보안 진단은 구동 중인 애플리케이션 패킷을 제어하는 **동적 진단 위주로 대부분 수행**됩니다. 소스코드 기반의 정적 진단을 가동할 때는 시스템 내부의 모든 입출력 포인트(Input/Output Point)를 전수 식별해 낸 다음, 데이터가 전송되고 처리되는 핵심 제어 키워드인 **`GET`**, **`SET`**, **`IN`**, **`OUT`**의 흐름을 추적하여 로직을 진단해야 합니다.
- **인코딩 분석 및 권한 검증:** 패킷 내부에 은닉되거나 변조된 파라미터가 포함되어 있을 때, 그것이 **Base64 인코딩** 형태인지 등을 정확히 파악하고 디코딩하여 대조해야 취약점 여부를 가려낼 수 있으며, 일반 권한 상승 취약점도 `Burp Suite` 프록시를 활용한 서버 사이드 체크 검증 과정을 필수로 거쳐야 합니다.
- **응답 값(Response) 증적의 압도적 중요성:** 취약점 증적 스크린샷을 확보할 때, 단순히 공격자가 변조된 파라미터를 전송한 '요청 값(Request)' 화면을 캡처하는 것은 가치가 낮습니다. 그보다 **악성 스크립트나 공격 구문이 시스템을 뚫고 들어가 실제 데이터베이스를 거쳐 튀어나온 '응답 값(Response)' 화면 및 브라우저 실행 화면의 스크린샷을 확보하는 것이 실무적으로 훨씬 치명적이며 중요**합니다.
- **보안 진단 프로세스 5단계 수립:** 표준 거버넌스는 **[보안진단 신청(담당자)  진단 수행(진단자)  취약점 조치(담당자)  이행점검 수행(진단자)  보안진단 완료]**의 5단계 파이프라인으로 움직여야 합니다. 진단 기준 치명도는 **`긴급`**, **`중요`**, **`일반`** 3단계 등급으로 이원화 관리하며, 기준 판정이 다소 애매모호한 취약점의 경우에는 컴플라이언스상 `8-1` 그룹으로 제외 분기하여 관리합니다.
- **심화 위협 및 명령어 침투 메커니즘:** 소프트웨어 패치 배포 전 빈틈을 타서 공격하는 **제로데이(Zero-Day) 취약점**의 위협 속성을 이해하고, 객체 복원 시 악성 로직을 기동시키는 **역직렬화(Deserialization)** 공격 원리와 공개 출처 인텔리전스인 **OSINT** 기법을 정립했습니다. 또한 SQL Injection을 통해 서버 내 시스템 명령어를 제어하는 **`xp_cmdshell`** 확장 프로시저의 침투 원리를 파악했습니다.


# 4. 아르고스 실전 DAST 가동 현황: Swagger 명세 주입 및 500 에러 오탐 분석

19일차에 확정한 동적 분석(DAST) 집중 전략에 따라, 실제 배포된 온데(Onde) 플랫폼(`onde.click`)을 대상으로 최초 자동화 스캔 테스트를 개시하여 패킷 레벨의 정/오탐 디깅을 수행했습니다.

### ① API 크롤링의 패러다임 전환: ZAP Spider의 한계와 Swagger 주입

- **레거시 Spider 크롤링의 한계 식별:** OWASP ZAP의 내장 크롤러인 'Spider' 기능만을 단독 실행하여 진단 경로를 수집하려 했으나, 사전 컨텍스트 정보가 제공되지 않으면 깊숙한 곳에 은닉된 API URL 구조를 세밀하게 찾아내지 못해 진단 성능이 크게 떨어지는 결함이 발견되었습니다.
- **Swagger.json 명세 분석 엔진 도입:** 이를 해결하기 위해 개발자가 구축한 전체 API들의 규격, 요청 경로, 입력값 타입 등을 한눈에 볼 수 있도록 자동 시각화해 주는 도구인 Swagger를 활용했습니다. 서버가 자동으로 빌드해 내는 **`swagger.json` 명세 파일을 추출하여 ZAP 스캔 엔진에 직접 주입**하는 파이프라인을 구축했습니다. 이 명세서 정보를 기반으로 스캐너가 전체 API URL 경로를 누락 없이 식별하고 추적하여 테스트 진단을 수행할 수 있었습니다.

### ② 500 에러 페이지 반환에 따른 버퍼 오버플로우 오탐지 분석

- **오탐 원인:** ZAP 스캐너에 `swagger.json`을 주입하여 동적 스캔을 진행한 결과, **총 466개의 취약점 의심 항목이 감지**되었습니다. 그러나 이 중 대다수가 '버퍼 오버플로우' 취약점으로 잘못 분류되는 오탐(False Positive) 현상이 나타났습니다.
- **메커니즘 분석:** 테스트 대상 서버가 허용되지 않은 비정상 입력값을 받았을 때 예외 처리가 작동하여 HTTP 500 에러를 응답하거나 오류 화면으로 리다이렉트 시켰는데, 스캐너는 이를 서버의 치명적인 오동작(컴파일 엔진 크래시 및 버퍼 오버플로우 성공)으로 잘못 인식한 것이었습니다.
- **대응 방안:** 단순 서버 오류 응답(500)이나 에러 템플릿 반환을 걸러내기 위해서는 스캔 엔진의 정밀한 규칙(Rules) 및 정책 설계가 매우 중요하며, 해당 에러 응답들을 효과적으로 구분해 낼 수 있는 설정을 정교화하여 스캔 엔진 단계에서 정/오탐을 개선해 나갈 예정입니다.

### ③ 오류 페이지 노출 취약점 검증 및 실제 패킷 분석

비정상적인 입력 주입 시 에러 핸들링 미흡으로 인해 서버의 디버그 예외가 화면에 유출되는 명백한 실전 취약점인 **"6-1. 오류 페이지를 통한 정보 노출 여부(Application Error Disclosure)"**를 찾아내고 패킷 검증을 완료했습니다.

# 5. Next Step: 오탐 필터링 규칙 정교화 및 SK 가이드라인 기반 룰셋 매핑 수정

최초 동적 스캔을 통해 대량의 원시 데이터와 오탐 원인, 그리고 실제 취약점 패킷을 명확히 확보했으므로, 차일 마일스톤부터는 스캔 데이터의 정밀도를 프로덕션 수준으로 튜닝하는 공정에 착수합니다.

- **ZAP 스캔 정책(Scan Policy) 정교화 튜닝:** ZAP 설정을 정교하게 튜닝하여 500 에러 등의 오탐을 필터링하고 실제 서버 예외와 오동작을 구별하는 차단 규칙 수립.
- **SK쉴더스 가이드라인 완벽 동기화:** 실제 SK 가이드라인 기준에 완벽하게 부합하는 취약점들만 정교하게 잡아낼 수 있도록 규칙을 수정 및 적용할 예정.
