# 1. 개요 및 공통 인프라 비동기·인증 풀 계층 명세

여러 브랜치에 분산되어 있던 개별 진단 엔진 슬롯들을 SK쉴더스 웹/API 취약점 진단 가이드라인 표준 아키텍처 위로 수렴하기 위해 공통 백엔드 인프라 계층을 대대적으로 하드닝했습니다. 이번 비교 구간은 `935383a` 커밋부터 HEAD인 `a87dbb9`까지의 총 17개 커밋 diff 사양을 명확히 반영합니다.

## ① 6-1 중심의 실시간 비동기(Async) 실행 및 취소(Cancel) 제어 프레임워크 수립 (`5f563f3`)

대용량 퍼징 주입 시 발생하는 게이트웨이 타임아웃(Gateway Timeout)을 원천 차단하고 런타임 제어권을 확보하기 위해 비동기 파이프라인을 전면 도입했습니다.

- **백엔드 비동기 통제 라우팅:** `POST /api/diagnosis/modules/6-1/run` 요청 인입 시 백엔드 라우터는 즉각 `202 Accepted` JSON 응답을 배출하고 백그라운드 스레드 워커를 구동합니다.
- **대시보드 실시간 폴링 구조:** 프론트엔드는 `GET /api/diagnosis/progress` 엔드포인트를 실시간 폴링(polling)하며 진행 상태 바를 유기적으로 동기화합니다.
- **스캔 취약점 즉시 취소 가드:** 사용자가 실행 중 버튼을 재클릭하면 `POST /api/diagnosis/cancel` 서비스가 트리거되어 내부 `DiagnosisCancelled` 예외를 배출합니다. 이를 통해 무거운 프로브 배칭 프로세스를 즉각 안전하게 중단(Prune)시키고, 중단 시점까지 수집된 부분 결함 증적 데이터만 `latest.yaml` 리포트에 안전하게 영속화 보존하도록 설계했습니다.

## ② 다중 권한 교차 주입용 자동 인증 패스 및 세션 풀 라이프사이클 관리 (`d291001`)

- **인증 요구 조건 매핑 자동화:** 엔드포인트별 인가 권한 요구 조건에 맞춰 가상 테스트 계정의 세션 토큰 조합을 자동으로 추출하고 매핑해 주는 `endpoint_auth_passes.py` 인프라 클래스(+298라인)를 신설했습니다.
- **JWT 만료 차단 자동 리프레시 엔진:** `auth_session_pool.py`(+139라인)가 백그라운드에서 상시 연동되어 세션 캐시 및 토큰의 만료 스탬프(`exp`)를 추적하며, 토큰이 소멸하기 직전 단계에서 의무적으로 리프레시 절차를 자율 전개합니다.
- **연동 모듈 대역:** `2-2, 3-2, 3-5, 3-6, 4-1, 4-2, 5-2, 6-1, 6-2, 7-1~7-4` 등 아르고스 코어 스캐너 파이프라인 전반에 의무적으로 바인딩되어 멀티 패스 권한 조회를 무결하게 보장합니다.

## ③ 로컬 vs Docker 환경 Probe URL 라우팅 규칙 통일 개편 (`a87dbb9`)

- **결함 현상 및 원인 분석:** 백엔드 도커 컨테이너 내부망 환경에서 triggers 기동 시 타깃 주소로 `localhost:8080`이 그대로 유입되면, 외부 호스트 서버가 아닌 컨테이너 자기 자신으로 루프백 라우팅되어 `Connection Refused` 에러와 함께 status가 100% 실패 처리되는 미탐 결함을 적발했습니다.
- **통일 구조 패치 조치:** `6-1/triggers.py` 소스코드 단 내부에 공통 치환 유틸 매퍼인 `probe_url()` 및 `frontend_gateway_path()` 라인을 수직 결합하여, `build_probe_request(..., probe_base_fn=probe_url)` 형태로 호출이 변환되도록 수정 완료했습니다. 이를 통해 웹 대시보드 및 리포트 화면상에는 `localhost` 형상을 완벽히 유지하되, 하부의 실질적인 HTTP 퍼징 트래픽만 Docker 호스트 대역으로 안전하게 전달되도록 매퍼를 정상화했습니다.

# 2. 입력값 검증 및 삽입 위협 통제 계층 전사 병합 (1-1, 1-2, 1-4, 1-6)

SK쉴더스 취약점 진단 가이드라인 최상단에 위치한 핵심 탐지 모듈 브랜치들을 하나의 소켓으로 병합 완수했습니다.

## ── 1-1. XSS / CSRF 공격 가능성 모듈 (PR #11 · `4937772`)

- **보완 조치 및 설계 명세:** `feat/xss&csrf-search-engine` 브랜치의 고도화 성과를 반영하여 `modules/1-1/` 하부의 리팩토링 및 2차 연동을 완수했습니다.
- **동적 검증 및 동작 흐름:** `openapi_utils.py`를 거쳐 추출된 정제 seed 주소를 기반으로, 익명 자산 및 `admin / seller / user` 다중 역할별 컨텍스트 세션 JWT를 ZAP 스캔 엔진에 동적으로 바인딩 투사합니다. 대규모로 확장된 `zap_runner.py`(+696라인)가 기동되어 크로스 사이트 스크립팅 및 요청 변조 위협을 스파이더 및 액티브 스캔 오케스트레이션으로 전개한 뒤, `zap_adapter.py`가 중복 findings `collapse` 압축 및 심각도 정규화 처리를 집행합니다.

## ── 1-2. 삽입 (Injection) 공격 가능성 모듈 (`7512bc9` 및 후속 커밋)

- **보완 조치 및 설계 명세:** 단순히 결과 유무만 YAML로 배출하여 가독성이 저하되던 기존 단독 구조를 전면 해체하고, `1-2/module.py` 내부에 `_save_json_report()` 서브 로직을 신설 이식했습니다.
- **동적 검증 및 동작 흐름:** 진단 가동 시 실서버에 치명적인 영향을 주는 타임 기반/에러 기반 인젝션 지표는 **'정탐(Confirmed/Verified)'** 트랙으로 완전 격리하고, 단순 구문 변조 에러나 논리식 참/거짓 분기 노이즈는 **'약한 정탐(Suspected)'** 레이어로 쪼개어 타임스탬프 기반 JSON 이력 파일(`report_{timestamp}.json`)로 병행 분리 저장하도록 판정 가드를 하드닝했습니다. 아울러 `1-2/docker-compose.yml` 사양을 독립 추가하여 외부 인프라 간섭 없이 ZAP + Target 단독 실행 디버깅이 가능하도록 구조를 정돈했습니다.

## ── 1-4. SSRF / File Inclusion 공격 가능성 모듈 (`05b6ab5` + PR 머지 `7c58582`)

- **보완 조치 및 설계 명세:** `payload_injector.py` 핵심 컴포넌트를 대규모 리팩토링(+231라인 net)하여 외부 URL 강제 참조(SSRF) 및 원격/로컬 파일 인클루전(RFI/LFI) 탐지 정확도를 극한으로 끌어올렸습니다. `search_engine`, `input_parser`, `inventory_bridge`, `zap_engine` 전반의 파이프라인이 정비되었습니다.
- **동적 검증 및 동작 흐름:** `discover_service.py`와 `verify_service.py` 간의 무결한 상호 영속성 인터페이스 트랙을 연결하여, 검증 단계에서 추출된 영속 데이터 샘플 지표가 유실 없이 1-4 검색 엔진 코어에 도달하도록 다리를 놓았습니다. OpenAPI 구조 해석 시 중첩 구조 객체 파라미터(`nested object param`) 해석 옵션 사양을 정교화 빌드했으며, 머지 과정에서 발생한 빈 verify skip 가드 조건과 reference sample 복원 로직의 충돌을 수동 정밀 조율하여 데이터 무결성을 확보했습니다.

## ── 1-6. 입력 값 크기 및 무결성 검증오류 모듈 (PR #9 · `7399024`)

- **보완 조치 및 설계 명세:** JangSeonguk1011 담당의 W-1-6 고속 퍼징 검색 엔진 프레임워크 전체 자산(+7,274라인 규모)을 `backend/diagnosis/modules/1-6/` 하위 소켓에 무결하게 포팅 임베드 완료했습니다.
- **동적 검증 및 동작 흐름:** 내부 구조에 `fuzzer.py`(파라미터 fuzzing 코어), `collector.py`(응답/증적 수집), `screenshot.py`(UI 자동 캡처), `zap_engine.py` 결합 제어선 및 Swagger 사양 해석용 모듈을 내장 탑재했습니다. 프론트엔드 단에 `DiagnosisG16RunOptions` 인터페이스 규격과 전용 시작 모달 다이얼로그 연동을 연계 마감하여 대시보드 상에서 완벽히 통제 가능하며, 실서버 검증을 위한 `docs/ONDE_INPUT_SEARCH_ENGINE_RUNBOOK.md` 운영 가이드를 통합 탑재했습니다.

## ── 2-2. 중요 정보 파일 다운로드 가능성 모듈 (런타임 크래시 패치 · `d291001`)

- **결함 현상 및 버그 교정:** 대시보드 연동 구동 시 `2-2/probes.py` 소스코드 단 내부에서 로컬 매핑 인자인 `current_auth`를 바인딩 변수로 참조하려다 스펙 불일치로 인해 `NameError`와 함께 시스템 500 내부 에러 예외를 발생시키며 전체 스캔 트랙이 파기되던 치명적인 버그를 포착했습니다.
- **조치 및 흐름 매핑:** 금일 자 리팩토링 트랙을 통해 변수 참조 관계를 전사 공통 세션 토큰 체계인 `auth` 명세 라인으로 전향 전치 수정 완료하여 500 구동 오류를 완벽하게 정상 복구하고 IDOR 프로브 연동 무결성을 정상 안착시켰습니다.

# 4. 전사 14개 진단 모듈 풀스택 대시보드 스캔형 패널 연동 명세

사용자 인터페이스 가독성과 리포트 신뢰도를 극대화하기 위해 기존의 날것으로 출력되던 상세 로그 뷰 포맷을 완전히 전면 폐기하고, **'통합 요약 지표 테이블 + 세부 페이로드 접기(Collapsible) 상세 패널'** 표준 레이아웃 구조로 프론트엔드 계층을 100% 전면 리팩토링 개편 완료했습니다. findings 스케일이 수백 건을 초과하더라도 브라우저 스크롤 락 및 성능 저하가 발생하지 않도록 비동기 지연 렌더링 가드인 `CollapsibleReportSection.tsx`를 도입했습니다.

## ── 대시보드 리포트 패널 라우팅 맵 확장 명세 (`DiagnosisReportPanel.tsx`)

- **`1-5` (오픈 리다이렉트):** 불필요한 카드 섹션을 소멸시키고 CORS 위협 지표를 직관적인 **한국어 요약 텍스트 명세**로 파싱 출력하도록 개편했습니다.
- **`5-2` (민감정보 PII):** 응답 트래픽 내에서 수집된 HTTP/body/2xx/401 통계 카운터 카운트 수치와 유출 속성을 일대일 대응 매핑한 `G52FindingsPanel`을 수립했습니다.
- **`2-2, 3-2, 3-4, 3-5, 3-6, 4-2` 계열:** `G22~G42` 전용 정형화 리포트 뷰 객체를 정밀 빌드하여, 디렉터리 변조 및 락아웃 임계치 통계를 계층형 구조로 정렬 관제합니다. 특히 `3-5` 모듈은 review finding 검출 상태에 따라 대시보드 상에 `warn` 상태 지표를 동적으로 배출합니다.
