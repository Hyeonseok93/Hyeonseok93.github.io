# 서론

> **"주간 멘토링에서 도출된 피드백을 기반으로 진단 대상인 여행 플랫폼 '온데(Onde)'의 기능 고도화와 관리자 격리 패치를 완벽히 반영했습니다. 이와 동시에 자동 진단 플랫폼 '아르고스(Argus)'의 개발 생산성과 유기적 결합도를 극대화하기 위해, 기존의 팀별 분업 방식에서 '1인 전 공정(End-to-End) 파이프라인 빌드' 체제로 전환 했습니다. 진단 범위를 정적 분석에서 100% 동적 분석 중심으로 압축하고 SK 가이드라인 명세를 촘촘히 보완하며 실전 진단 자동화의 기틀을 마련했습니다."**
>
> 멘토링 피드백 기반 온데 기능 고도화를 반영하고, 아르고스는 1인 End-to-End 파이프라인·DAST 전념·SK 가이드 28항목 동기화 전략으로 전환했습니다.

# 1. 진단 대상 '온데(Onde)' 플랫폼 멘토링 피드백 피처 반영 및 고도화 완수

2차 대면 멘토링에서 지적된 공격 표면 확대 및 비즈니스 로직 보완 요구사항을 완벽하게 수렴하여, 온데(Onde) 플랫폼의 대대적인 기능 추가 및 인터페이스 수정을 마감했습니다.

## ① 커뮤니티 기능 확장 및 회원가입 중복 검사 도입

- **리뷰 댓글 컴포넌트 고도화:** 여행기 리뷰 영역에 일반 댓글뿐만 아니라 인가된 사용자 간 비밀 소통이 가능하도록 **'비밀 댓글 기능'**을 추가 구축했으며, 본인이 작성한 피드를 유연하게 정정할 수 있도록 **'여행기 수정 기능'**을 이식 완료했습니다.
- **회원가입 검증 레이어 보완:** 회원가입 입력 폼 단계에서 이름, 전화번호, 닉네임, 나이 외에 식별자 오염을 차단하기 위한 **[중복 확인 기능]을 필수 추가**하여 회원 데이터의 무결성을 강화했습니다.
- **판매자 대시보드 신설:** 판매자가 매출의 발생 기전과 트렌드를 직관적으로 트래킹할 수 있도록 **어디서 매출이 발생하는지 정밀 집계해 주는 판매자 대시보드 패널**을 연동했습니다.

## ② 어드민 로그인 격리 및 인터랙션 최적화

- **독립형 관리자 로그인 페이지 신설:** 사용자 로그인 창과 공유되던 보안 취약 구조를 파괴하고, **관리자(Admin) 전용 격리 로그인 화면을 별도 추가**하여 물리적인 접근 통제 경계를 확립했습니다.
- **다운로드 경로 할당 제거:** 마이페이지 예약 내역 다운로드 시 특정 물리 절대 경로가 지정되어 크래시를 유발하거나 경로 조작 취약점을 주던 로직을 전면 제거하고 표준 브라우저 다운로드 스트림으로 전환했습니다.
- **토스트 알림 반응 속도 개선:** 사용자 행동 피드백의 기민함을 고조시키기 위해 전역 토스트(Toast) 메시지 노출 지속 시간을 기존 **3초에서 1초로 축소** 튜닝하여 쾌적한 UX를 완성했습니다.

## ③ 마이페이지 프로필 관리 및 정산 명세 세분화

- **사용자 · 판매자 마이페이지 구축 (이예린):** 일반 유저 및 파트너 셀러가 본인의 계정 보안을 직접 제어할 수 있도록 **프로필 관리 기능(전화번호, 닉네임, 비밀번호, 이메일 변경)**을 각각 독립형 컴포넌트로 완비했습니다.
- **정산 상세 정보 명세화 (홍지호):** 판매자 및 본사 관리자 정산 페이지에서 단순 총매출액 합산만 보여주던 불충분한 화면을 개편하여, 단건 예약별 상세 정보까지 깊숙이 인덱싱되도록 스키마 바인딩을 추가했습니다.
- **인프라 구성도 보완:** 도메인 분리 및 윈도우 가상 서버 결합에 맞추어 클라우드 인프라 토폴로지 구성도를 최신 배포 사양으로 업데이트했습니다.

# 2. 아르고스(Argus) 플랫폼 개발 전략의 전환

보안 진단 자동화 플랫폼 '아르고스(Argus)'의 아키텍처 싱크업 과정에서, 한정된 타임라인 내에 이기종 엔진 간의 세션 바인딩 결함을 지우고 유기적인 데이터 유효성을 확보하기 위해 **기획 및 개발 거버넌스 전략을 전격 수정**했습니다.

## ① 3개 팀 분리 개발에서 '1인 통합 파이프라인 빌드'로 전환

- **기존 방식의 한계 극복:** 당초 A팀(엔진), B팀(셀레니움), C팀(리포트)으로 분할하여 개별 모듈을 만드는 방식은 모듈 간 데이터 규격 조율 시 인터페이스 충돌 병목이 잦을 것으로 판단했습니다.
- **수정된 체제:** 각 전담 팀이 아르고스의 기본적인 공통 뼈대 구조(Skeleton)를 함께 의논하여 확정한 뒤, **한 명의 개발자가 스캔 엔진(A)  Selenium 자동 증거 수집(B)  최종 보고서 작성(C) 전체 파이프라인 공정을 동시 연속적으로 다이렉트 빌드**하는 체제로 전향했습니다. 스캔 엔진 작동과 셀레니움 증적 캡처가 단일 프로세스 컨텍스트 내에서 유기적으로 상호작용하도록 결합도를 극대화했습니다.

## ② 진단 스코프의 압축: 동적 분석(DAST) 전념

- **정적 분석 사양 제거:** 멘토링 특강 분석 결과를 반영하여, 실효성이 낮고 오탐율이 높은 소스코드 정적 분석(SAST/Semgrep) 연동 파이프라인을 과감히 제거했습니다.
- **동적 분석 집중 셋업:** 패킷 변조 및 실시간 파라미터 주입 공격의 PoC 무결성을 입증하기 위해, 오직 **동적 분석(DAST / OWASP ZAP) 파이프라인 및 응답 값 캡처 구현에 모든 진단 역량을 100% 집중**하기로 의사결정했습니다.

## ③ 리포팅 구조 슬림화 및 SK 가이드라인 동기화

- **보고서 2원화 정돈:** 기존 3개 분할(사용자, 판매자, 관리자) 발행 계획을 폐기하고, 실무 효율성을 위해 **(일반 사용자 / 판매자 통합 보고서) 1종**과 **(본사 관리자 전용 보고서) 1종**으로 보고서 아웃풋을 총 2개 구조로 명확히 정돈했습니다.
- **SK 가이드 룰셋 보완:** 아르고스 취약점 탐지 룰셋 모듈 내부를 전수 검사하여, 국내 표준인 **SK쉴더스 Web/API 취약점 진단 가이드라인**의 8개 그룹 28개 항목 중 매핑이 빠져 있던 세부 명세 항목들을 식별하고 완벽하게 채워 넣는 보완 작업을 마쳤습니다.

# 3. Next Step: 아르고스 뼈대 구조 확정 및 온데(Onde) 대상 최초 취약점 스캔 가동

진단 대상의 멘토링 피드백 기능 개발이 완벽하게 마감되고, 아르고스의 1인 파이프라인 집중 전략이 수립됨에 따라 차일부터는 본격적인 자동화 스캔 가동 단계로 돌입합니다.

- **아르고스 뼈대 구조(개발 방법) 의논 후 확정:** 데이터 파싱 규격과 단일 스레드 내 ZAP API - Selenium 캡처 핸들러 간의 실행 순서를 정의하는 뼈대 설계를 최종 마감.
- **Web/API 취약점 최초 진단(Initial Scanning) 개시:** 뼈대 구조가 안착하는 즉시 실서버에 배포 완료된 온데 플랫폼(`onde.click`)을 스캔 범위로 설정하여 최초 취약점 진단
- **최초 진단 보고서(Initial Report) 초안 빌드:** 동적 분석 엔진을 통해 수집된 8개 그룹 28개 항목 로우 데이터와 Selenium 응답 값 증적 스크린샷을 병합하여 아르고스 플랫폼이 찍어내는 첫 번째 최초 진단 보고서 출력 테스트 수행.
