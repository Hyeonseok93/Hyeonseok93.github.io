>

# 1. 온데(Onde) 멘토링 피드백 반영

2차 대면 멘토링에서 나온 공격 표면 확대와 비즈니스 로직 보완 요구사항을 반영해, 온데(Onde) 플랫폼의 기능과 인터페이스를 수정했습니다.

## ① 커뮤니티 기능 확장 및 회원가입 중복 검사 도입

- **리뷰 댓글 기능 보완:** 여행기 리뷰 영역에 일반 댓글뿐 아니라 인가된 사용자끼리 비밀스럽게 대화할 수 있는 **'비밀 댓글 기능'**을 추가했고, 본인이 작성한 피드를 수정할 수 있는 **'여행기 수정 기능'**도 넣었습니다.
- **회원가입 검증 레이어 보완:** 회원가입 입력 폼 단계에서 이름, 전화번호, 닉네임, 나이 외에 식별자 오염을 차단하기 위한 **[중복 확인 기능]을 필수 추가**하여 회원 데이터의 정확성을 강화했습니다.
- **판매자 대시보드 추가:** 판매자가 매출의 발생 기전과 트렌드를 직관적으로 트래킹할 수 있도록 **어디서 매출이 발생하는지 집계해 주는 판매자 대시보드 패널**을 연동했습니다.

## ② 어드민 로그인 격리 및 인터랙션 최적화

- **독립형 관리자 로그인 페이지 추가:** 사용자 로그인 창과 공유하던 구조를 분리하고, **관리자(Admin) 전용 로그인 화면을 따로 추가**해 접근 경계를 나눴습니다.
- **다운로드 경로 할당 제거:** 마이페이지 예약 내역 다운로드 시 특정 물리 절대 경로가 지정되어 크래시를 유발하거나 경로 조작 취약점을 주던 로직을 제거하고 표준 브라우저 다운로드 스트림으로 전환했습니다.
- **토스트 알림 반응 속도 개선:** 전역 토스트(Toast) 메시지 노출 시간을 기존 **3초에서 1초로 줄였습니다.**

## ③ 마이페이지 프로필 관리 및 정산 명세 세분화

- **사용자 · 판매자 마이페이지 구축 (백엔드):** 일반 유저 및 파트너 셀러가 본인의 계정 보안을 직접 제어할 수 있도록 **프로필 관리 기능(전화번호, 닉네임, 비밀번호, 이메일 변경)**을 각각 독립형 컴포넌트로 완비했습니다.
- **정산 상세 정보 명세화:** 판매자 및 본사 관리자 정산 페이지에서 단순 총매출액 합산만 보여주던 화면을 개편하여, 단건 예약별 상세 정보까지 인덱싱되도록 스키마 바인딩을 추가했습니다.
- **인프라 구성도 보완:** 도메인 분리 및 윈도우 가상 서버 결합에 맞추어 클라우드 인프라 토폴로지 구성도를 최신 배포 사양으로 업데이트했습니다.

# 2. 아르고스(Argus) 모듈 별 1인 개발 파이프라인

보안 진단 자동화 플랫폼 '아르고스(Argus)'의 구조를 맞추는 과정에서, 제한된 일정 안에 엔진 간 세션 연결과 데이터 형식을 맞추기 위해 개발 방식을 바꿨습니다.

## ① A/B/C 분업에서 모듈 별 1인 개발 파이프라인으로

- **기존 방식의 한계:** 당초 A팀(엔진), B팀(Selenium), C팀(리포트)으로 분할하여 개별 모듈을 만드는 방식은 모듈 간 데이터 규격 조율 시 인터페이스 충돌 병목이 잦을 것으로 판단했습니다.
- **수정된 방식:** 팀이 아르고스의 기본 공통 뼈대 구조(Skeleton)를 함께 정한 뒤, **한 명의 개발자가 스캔 엔진(A) → Selenium 자동 증거 수집(B) → 최종 보고서 작성(C)까지 이어서 개발**하는 모듈별 1인 개발 방식으로 바꿨습니다. 스캔 엔진과 Selenium 증적 캡처가 한 프로세스 안에서 자연스럽게 연결되도록 했습니다.

## ② 진단 스코프의 압축: 동적 분석(DAST) 전념

- **정적 분석 사양 제거:** 멘토링 특강 분석 결과를 반영하여, 실효성이 낮고 오탐율이 높은 소스코드 정적 분석(SAST/Semgrep) 연동 파이프라인을 과감히 제거했습니다.
- **동적 분석 집중 셋업:** 패킷 변조 및 실시간 파라미터 주입 공격의 PoC 정확성을 입증하기 위해, 오직 **동적 분석(DAST / OWASP ZAP) 파이프라인 및 응답 값 캡처 구현에 진단 역량을 집중**하기로 했습니다.

## ③ 리포팅 구조 슬림화 및 SK 가이드라인 동기화

- **보고서 2원화 정돈:** 기존 3개 분할(사용자, 판매자, 관리자) 발행 계획을 폐기하고, 실무 효율성을 위해 **(일반 사용자 / 판매자 통합 보고서) 1종**과 **(본사 관리자 전용 보고서) 1종**으로 보고서 아웃풋을 총 2개 구조로 명확히 정돈했습니다.
- **SK 가이드 룰셋 보완:** 아르고스 취약점 탐지 룰셋 모듈 내부를 전수 검사하여, 국내 표준인 **SK쉴더스 Web/API 취약점 진단 가이드라인**의 8개 그룹 28개 항목 중 매핑이 빠져 있던 세부 명세 항목들을 식별하고 채워 넣는 보완 작업을 마쳤습니다.

# 3. 다음 단계: 아르고스 기본 구조 확정과 온데(Onde) 첫 취약점 스캔

온데 멘토링 피드백 반영을 마감하고, 아르고스 모듈 별 1인 개발 파이프라인 방향을 잡은 뒤, 차일부터는 자동화 스캔 스캔 단계로 넘어갑니다.

- **아르고스 뼈대 구조(개발 방법) 의논 후 확정:** 데이터 파싱 규격과 단일 스레드 내 ZAP API - Selenium 캡처 핸들러 간의 실행 순서를 정의하는 뼈대 설계를 마무리.
- **Web/API 취약점 최초 진단(Initial Scanning) 개시:** 뼈대 구조가 자리 잡음하는 즉시 실서버에 배포 완료된 온데 플랫폼(`onde.click`)을 스캔 범위로 설정하여 최초 취약점 진단
- **최초 진단 보고서(Initial Report) 초안 빌드:** 동적 분석 엔진을 통해 수집된 8개 그룹 28개 항목 로우 데이터와 Selenium 응답 값 증적 스크린샷을 병합하여 아르고스 플랫폼이 찍어내는 첫 번째 최초 진단 보고서 출력 테스트 수행.
