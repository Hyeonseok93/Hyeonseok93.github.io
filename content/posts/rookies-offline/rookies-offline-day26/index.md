# 서론

> **"이전 최초 취약점 진단 단계에서 식별된 보안 결함들을 완벽히 수정하고 안전성을 재입증하기 위해, 일반 사용자 및 판매자 영역을 대상으로 대대적인 '최초 이행 점검(Re-inspection)'을 집행했습니다[cite: 549, 665, 687]. 이번 상편 일지에서는 전사 28개 표준 마스터 항목 중 절반에 달하는 1-1 입출력 검증부터 3-6 백업 파일 존재 여부까지의 조치 및 동적 패킷 검증 내역을 다룹니다[cite: 575, 588, 682, 687]. 전역 필터링 레이어 도입과 아키텍처 체질 개선을 통해 보안 구멍을 완벽히 차단하고 '이행 완료' 판정을 획득한 상세 기술 명세를 요약 공유합니다."**
>
> 최초 진단에서 식별된 결함을 보완한 뒤 일반 사용자·판매자 영역에 대해 이행 점검을 집행했습니다. 1-1부터 3-6까지의 조치 및 검증 내역을 다룹니다.

# 1. 점검 개요 및 총평

기 수행된 최초 보안 진단에서 도출된 다양한 보안 취약점 항목들의 조치 적정성을 검증하고 잔존 위협을 최소화하기 위해 정밀 이행 점검에 착수했습니다. 이번 점검은 코드 분석, 서버 설정 검증, 그리고 프록시 도구(`Burp Suite`)를 활용한 동적 우회 시뮬레이션을 다각도로 결합하여 집행되었습니다.

검증 결과, 파편화되어 있던 비즈니스 로직 단에 공통 검증 모듈 및 전역 예외 처리 체계가 견고하게 이식되었음을 확인하였으며 점검 범위 내 모든 항목에 대해 최종 **'이행 완료(적합)'** 판정을 획득하는 성과를 거두었습니다.


# 2. 입/출력 값 검증 부재 영역 (1-1 ~ 1-6)

### ── 1-1. XSS / CSRF 공격 가능성 (중요 - 이행완료)

- **보완 조치:** 기존 소스코드 내 문자열 정제 로직 누락으로 발생하던 Stored XSS 위협을 차단하기 위해 백엔드 단에 Hibernate `PRE_INSERT` 및 `PRE_UPDATE` 전역 리스너 아키텍처를 도입했습니다. JPA 계층에서 저장·수정되는 모든 엔티티의 String 필드가 `InputSanitizer` 필터를 일괄적으로 강제 거치도록 조치했습니다. 이를 통해 `<script>`, `<style>` 태그 및 위험 이벤트 핸들러가 저장 직전에 원천 소멸됩니다.
- **동적 검증:** 여행기 글쓰기, 댓글, 프로필 수정, 상품 등록, 회원가입 등 전체 API 파라미터에 악성 스크립트 구문을 강제 주입하여 등록 테스트를 수행한 결과, 요청 본문에는 스크립트가 포함되어 전송될 수 있으나 저장 이후에는 악성 코드가 완전히 제거된 채 일반 텍스트만 안전하게 반환되어 브라우저 내 실행이 차단됨을 확인했습니다. 본 점검 범위에서 최종 이행 적합 판정을 획득했습니다.

### ── 1-2. 삽입 (Injection) 공격 가능성 (중요 - 이행완료)

- **보완 조치:** 기존 여행기 조회 API(`GET /user-api/api/v1/posts`)의 `status` 파라미터가 Native SQL 문자열에 직접 연결되어 UNION SELECT 쿼리가 우회 실행되던 구조적 결함을 전면 리팩토링했습니다. 문자열 결합 방식의 `PostRepositoryCustomImpl`을 제거하고, status를 `PostStatus` enum 타입으로 수신한 뒤 JPQL Named Parameter(`:status`) 매개변수 바인딩 처리를 거치도록 아키텍처를 정형화했습니다.
- **동적 검증:** `status` 변수 영역에 UNION SELECT 등의 SQL 인젝션 공격 페이로드를 주입하여 송신한 결과, enum 데이터 형식 전환 단계에서 원천 거부되어 비인가 데이터 유출 없이 HTTP 400 에러와 함께 안전하게 차단됨을 확인했습니다.

### ── 1-3. 파라미터 값 및 히든(hidden) 필드 조작 가능성 (중요 - 이행완료)

- **Case 1 (결제 요청 금액 변조 우회 결제):** 항공권 예약 등록 API 호출 시 클라이언트가 던지는 `totalPrice` 인자를 서버가 그대로 수용하던 논리적 결함을 차단했습니다. 백엔드 `FlightService.bookSeat()` 로직 내부에서 좌석 재고의 `basePrice` 정가 정보와 승객 수를 곱해 결제 단가를 서버 단에서 직접 강제 산출하도록 변경하고, 일치하지 않으면 `ValidationException` 예외로 요청을 즉각 거부하도록 조치했습니다. 임의의 낮은 금액(10원) 변조 패킷 송신 시 즉각 HTTP 400 오류를 배출하며 거래 생성을 반려합니다.
- **Case 2 (음수 마일리지 파라미터 조작):** 결제 사전 등록 DTO 객체 내부의 `usedMileage` 필드에 `@Min(0)` 및 `@Max` Bean Validation 제약 조건을 명시적으로 바인딩하고 보유 잔액 초과 여부를 추가 검증하도록 조치했습니다. 사용자가 입력단에 음수 값을 강제 삽입하여 패킷을 우회 송신하더라도, 컨트롤러 진입 전 단계인 `@Valid` 검증 계층에서 안전하게 차단하여 HTTP 400 오류를 반환함을 실증했습니다.

### ── 1-4. SSRF / File Inclusion 공격 가능성 (중요 - 이행완료)

- **Case 1 (템플릿 경로 조작 로컬 파일 유출 - LFI):** 통합 정산서 PDF 리포트 생성 API 호출 시 `template` 변수값을 파일 경로로 직접 수용하던 위험한 코드를 전면 전면 삭제했습니다. `IntegratedReportRequest` 내에 오직 `verification`과 `business` 단어 포맷만 수용되도록 강력한 `@Pattern` 검증 및 화이트리스트 enum 변환 구조를 수립하여 경로 조작 문자 주입 시 즉각 차단됩니다.
- **Case 2 (로고 주소 조작 내부망 요청 위조 - SSRF):** 요청 DTO 파라미터에서 클라이언트가 목적지 URL을 직접 제어할 수 있었던 `logoUrl` 필드를 기술적으로 완전히 영구 삭제 처리했습니다. PDF 헤더 로고 처리는 `BrandLogoLoader` 컴포넌트를 이용해 서버 시스템 클래스패스 내부에 번들링 보관된 안전한 정적 이미지 리소스 자원만 로드하도록 변경하여 내부망 메타데이터 접속 위협을 근본적으로 소멸시켰습니다.

### ── 1-5. 검증되지 않은 리다이렉트와 포워드 (일반 - 이행완료)

- **보완 조치:** 임시 개발용 호스트 규칙이 운영 서버 환경에 잔재하여 동일출처정책(SOP)을 무력화할 수 있었던 CORS 보안 설정을 수정했습니다. `CorsOriginProperties` 구조를 개편하여 공식 운영 프로필 환경(`prod`)에서는 `allow-localhost=false` 구동을 기본값으로 강제하고 공식 운영 도메인 패턴(`https://*.onde.click`)만 활성화되도록 분리 격리했습니다.
- **동적 검증:** `CorsOriginStartupValidator` 컴포넌트를 탑재하여 prod 프로필에서 localhost 허용 규칙이 활성화되면 애플리케이션 기동 자체를 즉각 중단(Shutdown)시키도록 통제를 강화했으며, curl 툴을 통한 위조 오리진 OPTIONS 요청 타격 시 차단 처리가 정상 작동함을 최종 입증했습니다.

### ── 1-6. 입력 값 크기 및 무결성 검증오류 (중요 - 이행완료)

- **보완 조치:** 회원가입, 프로필, 게시글, 항공 예약, 결제 요청, 댓글, 매물 등록 및 검색 쿼리 스트링 등 전사 파라미터 전반의 수치 범위 및 데이터 길이 제한 장치 누락 결함을 조치하기 위해 `ValidationLimits` 통제 명세판을 수립하고 상·하한 필터를 일제히 적용했습니다.
- **동적 검증:** 변조 스캐너를 통해 DB 컬럼 허용 상한 임계치를 초과하는 대용량 더미 데이터 및 포맷이 꼬인 필드를 인젝션하여 전송한 결과, 최상위 500 내부 오류를 뿜지 않고 `GlobalExceptionHandler` 전역 에러 제어기에 포획되어 안전하게 규격화된 한글 검증 메시지와 함께 HTTP 400 Bad Request로 안전하게 하드닝 처리됨을 전수 확인했습니다.


# 3. 취약한 파일 처리 영역 (2-1 ~ 2-2)

### ── 2-1. 악성코드파일 업로드 (중요 - 이행완료)

- **보완 조치:** 파일 업로드 비즈니스 로직 처리 시 확장자 및 MIME 검증 장치가 결여되어 저장형 XSS 및 RCE(원격 코드 실행)의 발판이 될 수 있었던 위협을 조치하기 위해 공통 모듈인 `ImageUploadValidator` 컴포넌트를 신설 전방 배치했습니다. JPG, PNG, WEBP 확장자 화이트리스트 정책 선언과 더불어 파일 헤더 매직 바이트(Magic Byte)를 검사해 실제 파일 내부 형식을 강제 검증하는 이중 방어 메커니즘을 이식했습니다.
- **동적 검증:** 파일명 내 경로 조작 문자(`../` 등) 필터링과 함께 `MultipartInputValidator` 연동을 통해 파일당 10MB 크기 제한 및 업로드 용량 한도를 설정했습니다. Burp Suite 도구로 이미지 파일 뒤에 우회 스크립트를 은닉 주입하거나 파일명을 `evil.html`로 위장 전송한 변조 시도 모두 "허용되지 않은 파일 확장자입니다" 검증 예외와 함께 HTTP 400번으로 원천 거부 차단됨을 실증 마감했습니다.

### ── 2-2. 중요 정보 파일 다운로드 가능성 (중요 - 이행완료)

- **보완 조치:** 통합 예약 명세서 리포트 생성 과정에서 공격자가 `template` 파일 경로 지정 인자 레이어에 Path Traversal 구문을 인젝션하여 서버 내부 기밀 파일 자산을 다운로드 PDF 출력물에 노출 및 유출시킬 수 있었던 결함을 전면 통제 패치했습니다. 앞서 [1-4] 지표와 연계하여 클라이언트가 서버 내부의 실제 물리 디렉터리 경로를 명시적으로 파싱 주입할 수 없도록 입력 구조를 완전 폐시 조치했습니다.
- **동적 검증:** 오직 사전에 시스템에 규정된 템플릿 코드 식별자 명세(`verification`, `business`) 화이트리스트 변수만 수용하도록 엄격화하고, `../` 기호 조합 데이터 강제 전송 시 `ValidationException` 분기망에 걸려 중요 설정 정보 파일(`.env` 등)이 외부로 절대 유출되지 않도록 구조적 안전 장치를 구축했습니다.


# 4. 취약한 접근 통제 관리 영역 (3-1 ~ 3-6)

### ── 3-1. 패스워드 정책 유무 및 반영 여부 (일반 - 이행완료)

- **보완 조치:** 단순 수치나 영문 조합만으로 계정이 생성되어 무차별 사전 대입 공격(Brute Force)의 제물이 될 수 있었던 회원가입 계정 생성 취약점을 소멸시키기 위해 core-module 내부에 `PasswordPolicy` 규칙 엔진 및 `@ValidPassword` 사용자 정의 Bean Validation 속성을 신설 적용했습니다.
- **동적 검증:** 일반 가입 시 영문·숫자·특수문자 중 3종 이상 조합 시 최소 8자 이상, 2종 조합 시 최소 10자 이상의 복잡도 길이를 강제하고 약한 비밀번호 블랙리스트와 공백 문자를 서버단에서 사전 거부하도록 설계했습니다. 가입 단계에서 정책 미달 암호 주입 패킷 전송 시 정책 안내 메시지와 함께 HTTP 400으로 안전하게 파기 처리됨을 확인했습니다.

### ── 3-2. 인증 실패 횟수 제한 (일반 - 이행완료)

- **보완 조치:** 패스워드 검증 실패를 무한정 허용하여 계정 하이재킹 위협을 유발했던 로그인 API 보안을 하드닝하기 위해 members 마스터 테이블 내부에 `failed_login_attempts` 카운터 및 `login_locked_until` 타임스탬프 스키마 컬럼을 추가 배치했습니다. 백엔드에 `LoginAttemptService`를 연동하여 비밀번호 불일치 실패 카운트가 연속 5회를 초과 판정되는 즉시 계정 로그인을 30분간 임시 잠금 처리하도록 조치했습니다.
- **동적 검증:** 변조 자동화 공격 툴을 기동해 동일 이메일 주소로 오답 패킷을 연속 타격해 본 결과, 5회 실패 이후 6번째 요청부터는 올바른 정상 정답 비밀번호 데이터를 가로채 송신하더라도 세션 통제를 선검사하여 접근을 전면 거부하고 HTTP 423 잠금 오류 코드를 엄격히 반환함을 입증 완료했습니다.

### ── 3-3. 계정 정보 파악 가능성 (일반 - 이행완료)

- **보완 조치:** 비인증 사용자 대상의 댓글 목록 API 조회 응답 내에 작성자의 실제 데이터베이스 시퀀스 고유 식별자인 `memberId` 키가 평문 유출되던 계정 파악 결함을 조치했습니다. 응답 규격판인 `CommentDto` 매핑 구조를 변경하여 내부 memberId 변수 항목을 완전히 제거하고, 오직 가시적인 화면 노출용 `authorName` 정보와 로그인 사용자 본인 여부 판단용 `isMine` 데이터만 한정 반환하도록 데이터 명세를 최소화 정돈했습니다.
- **동적 검증:** 작성자 표시명은 닉네임이 없을 때 식별자가 강제 조합되지 않도록 "익명" 상수로 전역 예외 처리했으며, 프록시 도구로 응답 패킷 본문 JSON 구조를 파싱 추적한 결과 memberId 키 자체가 영구 결여되어 유출 경로가 차단되었음을 확인했습니다.

### ── 3-4 ~ 3-6. 관리자 도메인 격리, 검색 노출 방지 및 백업 자산 전수 진단 현황 (양호/완료)

- **3-4. 관리자 페이지 분리 여부 (이행완료):** 사용자 서비스와 어드민 채널을 물리 격리한 것에 더해, 무작위 대입을 방지하고자 예측하기 쉬운 레거시 어드민 접속 서브도메인을 전면 폐기하고 난독화 세그먼트 규칙을 부여한 새로운 전용 포탈 주소(`https://rookies.onde.click/onde-entry-8k2p`) 체계로 변경을 완료했습니다. 외부 비인가자가 구 경로인 `/admin` 주소로 강제 진입 시 Nginx 프록시 및 프론트 단에서 HTTP 404 Not Found로 커넥션을 원천 파기시킵니다.
- **3-5. 검색엔진 정보 노출 가능성 (이행완료):** 공개 상용 루트 경로에 중요 도메인 디렉터리 접근 수집 제한 정책판인 `robots.txt`를 수립하고, 비공개 권한 포탈 및 파트너 셀러 페이지(/seller) 접근 통제 시 브라우저 헤더 단 내에 `<meta name="robots" content="noindex, nofollow">` 수집 거부 태그가 동적으로 강제 삽입·작동되도록 연동 설정을 완비했습니다.
- **3-6. 백업 파일 및 테스트 파일 존재 여부 (양호):** 배포 형상 관리 도중 도출될 수 있는 소스 흔적 및 임시 백업본 파일 존재 여부를 가려내기 위해 60대 다중 추측성 페이로드 사전 파일 셋을 Burp Intruder 도구에 실어 전수 무작위 대입을 타격했습니다. 확인 결과 유효한 은닉 자산 경로는 노출되지 않았으며 전체 미존재 경로에 대해 404 또는 안전한 예외 제어 가드가 유효하게 통합 작동 중임을 입증했습니다.
